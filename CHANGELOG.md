# История изменений

Проект состоит из двух связанных компонентов, у каждого своя нумерация:

- **`luci-app-podkop-bot`** — LuCI-интерфейс (нумерация `0.16.x`).
- **`install.sh`** — унифицированный установщик бота (нумерация `2.x` / `1.x`),
  поставляется внутри LuCI-пакета в `/usr/lib/podkop_bot/install.sh`.

Метки:

- **[Планово]** — сделано по дорожной карте ТЗ (срез, запланированная фича).
- **[Баг]** — внеплановое исправление, всплывшее на железе или при проверке.
- **[Долг]** — известная проблема, отложенная сознательно.

### 0.16.78 — Кеш status: сброс при ensure_mixed_proxy + честные границы
- **[Баг]** ensure_mixed_proxy не сбрасывал кеш status, а mixed_proxy state входит
  в status-вывод → Обзор до 5с показывал старое состояние после включения Mixed
  Proxy. Теперь кеш сбрасывается после изменения.
- **[Прим.] Риски устаревания кеша status (TTL 5с), честно:**
  1. Статичные данные (variant, версии podkop/sing-box/bot, pkg_manager) — 5с
     неважно, меняются только при переустановке (не через приложение).
  2. Изменения ЧЕРЕЗ приложение (install/update/uninstall, start/stop службы,
     ensure_mixed_proxy) — кеш сбрасывается, статус свежий.
  3. Внешнее изменение статуса бота (краш, OOM, stop из SSH, watchdog) — Обзор
     отстаёт до 5с. Это неизбежная граница кеша; но Обзор и так поллит раз в 5с,
     поэтому кеш НЕ добавляет задержки сверх интервала поллинга.
  4. set_tier1_port/set_uci_field/fallback_crud меняют поля, которых НЕТ в
     status-выводе — кеш их не затрагивает, рассинхрона нет.
- **[Отказ]** method_status оставлен на кеше (не сделан «всегда свежим»): Обзор
  поллит раз в 5с, кеш TTL 5с не ухудшает свежесть сверх интервала поллинга, зато
  экономит 12 opkg-сканов на каждый поллинг. Свежий status вернул бы нагрузку.
### 0.16.77 — Производительность Runtime: один jq-проход по sing-box config
- **[Планово]** runtime_sections: карта port→listen_ip строится ОДНИМ jq-проходом

- **[Прим.]** Кеш install.sh status (TTL 5с) уже был с 0.16.74 — «два status на
  вход Runtime» из ревью на деле один реальный запуск + один из кеша.
- **[Отказ]** runtime_bootstrap (объединить sections+probe в один RPC) не делаю:
  с кешем status второй вызов уже дешёвый, а метод дублировал бы логику. Фронтовый

  дублирования. wizard_env (лёгкий status) — по той же причине: кеш уже ускорил.

### 0.16.74 — Производительность: кеш status (меньше запусков install.sh)
- **[Планово]** Кеш результата install.sh --action status (TTL 5с). Причина: каждый
  status заново парсит весь install.sh и делает ~12 `opkg list-installed` сканов
  (медленно на opkg). 8 rpc-методов зовут status, при открытии вкладки набегало
  несколько запусков подряд → лаг Wizard/Обзора. Теперь в пределах 5с — один
  запуск, остальные из кеша. Кеш сбрасывается при install/update/uninstall и
  start/stop службы (статус всегда актуален после изменений).
- **[Прим.]** Это оптимизация по анализу кода (12 opkg-сканов на status, 8 точек
  вызова). Локально замерить не могу (нет opkg в песочнице) — подтверждается
  замером на роутере: time sh install.sh --action status. Если один status идёт
  секунду+, кеш убирает N-кратный множитель при открытии вкладок.

### 2.4.0 — [Баг] `msg()` съедал подстановки
Ветки `msg` печатали формат как `printf "литерал-с-%s"`, поэтому `$(msg X)`

  /proc/PID/cmdline` при исчезновении процесса в момент чтения. Чтение обёрнуто
